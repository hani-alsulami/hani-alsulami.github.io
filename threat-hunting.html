Threat Hunting

How I hunt, and the log of hunts run so far — each one starting as a hypothesis and ending as either a validated detection or a documented dead end.

My hunting playbook

The process behind every entry in the hunt log below — from a hunch to a shipped detection. Numbered because the order matters: skip a step and the finding won't survive review.

01

Form the hypothesis

Start from a specific, falsifiable claim tied to attacker behavior — not "look for anomalies." Ground it in a technique we haven't proven we'd catch, a gap flagged by recent threat intel, or a near-miss from a real incident.

MITRE ATT&CK Navigatorthreat intel feedspast incident reviews
02

Map the data

Identify exactly which log sources and fields would prove or disprove the hypothesis, and confirm they're actually being collected — with enough retention and fidelity — before writing a single query. If the telemetry isn't there, that gap gets logged and fixed before the hunt is trusted.

EDR/XDR telemetrySIEM index catalognetwork (NDR) logs
03

Build and scope the query

Write the narrowest query that could surface the behavior, then widen deliberately. Every hunt starts scoped to a small time window and a known-risky asset group before going environment-wide.

SPL / KQLSigma rulesSIEM correlation pivots
04

Baseline normal vs. anomaly

Every query is only as good as the baseline it's compared against. Define what "normal" looks like for that behavior first, so a hit means something before it ever reaches an analyst.

allow-listsknown-good traffic samplespeer review
05

Document the finding

Every hunt gets a short record regardless of outcome: hypothesis, data used, result, and false-positive rate. A hunt that found nothing is still worth logging — it closes a gap in what's been verified rather than just assumed safe.

hunt trackerATT&CK technique tagging
06

Hand off to detection engineering

A hunt that proves valuable doesn't stay a manual query — it graduates into a tuned, owned detection rule with a documented response path, so the next occurrence is caught automatically, not re-discovered from scratch.

correlation rulescase managementon-call runbook

T1110.003Password Spraying

Low-and-slow spraying against service accounts

HypothesisAn attacker is spraying passwords against AD service accounts slowly enough to stay under lockout thresholds — a high-value path since service accounts often lack MFA.
Data sourceAD authentication logs, SIEM correlation, cloud app access logs
MethodFlag single sources hitting multiple service accounts with failed logons outside normal maintenance windows.
Validated · detection rule deployed
T1539Steal Web Session Cookie

Session-cookie theft bypassing MFA

HypothesisAn attacker steals an active session cookie via a browser-based infostealer after a phishing click, then reuses it to bypass MFA entirely.
Data sourceEmail security gateway, endpoint telemetry, identity provider session logs
MethodFlag browser-to-script-host execution and session tokens reused across multiple devices or geographies in a short window.
In progress · tuning
T1003.001LSASS Memory

Credential dumping via LSASS access

HypothesisAn unrecognized or unsigned process reads LSASS memory to extract cached credentials without tripping AV signatures.
Data sourceEDR process-access telemetry, behavioral alerting
MethodBaseline which processes legitimately touch LSASS, then flag any unlisted or unsigned process requesting high-privilege access.
Planned · pending expanded telemetry
T1003.006DCSync

AD replication abuse for hash extraction

HypothesisA non-domain-controller account requests AD replication data, effectively extracting password hashes for arbitrary users.
Data sourceAD security event logs, allow-list of known domain controllers and sync accounts
MethodAlert on any replication request from an account outside the approved allow-list — near-zero false-positive once tuned.
Planned
