Threat Hunting

How I hunt, and the log of hunts run so far — each one starting as a hypothesis and ending as either a validated detection or a documented dead end.

My hunting playbook

The process behind every entry in the hunt log below — from a hunch to a shipped detection. Numbered because the order matters: skip a step and the finding won't survive review.

01

Form the hypothesis

Start from a specific, falsifiable claim tied to attacker behavior — not "look for anomalies." Ground it in a technique we haven't proven we'd catch, a gap flagged by recent threat intel, or a near-miss from a real incident.

MITRE ATT&CK Navigatorthreat intel feedspast incident reviews
02

Map the data

Identify exactly which log sources and fields would prove or disprove the hypothesis, and confirm they're actually being collected — with enough retention and fidelity — before writing a single query. If the telemetry isn't there, that gap gets logged and fixed before the hunt is trusted.

EDR/XDR telemetrySIEM index catalognetwork (NDR) logs
03

Build and scope the query

Write the narrowest query that could surface the behavior, then widen deliberately. Every hunt starts scoped to a small time window and a known-risky asset group before going environment-wide.

SPL / KQLSigma rulesSIEM correlation pivots
04

Baseline normal vs. anomaly

Every query is only as good as the baseline it's compared against. Define what "normal" looks like for that behavior first, so a hit means something before it ever reaches an analyst.

allow-listsknown-good traffic samplespeer review
05

Document the finding

Every hunt gets a short record regardless of outcome: hypothesis, data used, result, and false-positive rate. A hunt that found nothing is still worth logging — it closes a gap in what's been verified rather than just assumed safe.

hunt trackerATT&CK technique tagging
06

Hand off to detection engineering

A hunt that proves valuable doesn't stay a manual query — it graduates into a tuned, owned detection rule with a documented response path, so the next occurrence is caught automatically, not re-discovered from scratch.

correlation rulescase managementon-call runbook
Hunt log

Hunts I've designed and run using the playbook above — hypothesis, telemetry, method, and outcome. Environment specifics, exact query logic, and tooling names are deliberately generalized.

T1110.003Password Spraying

Low-and-slow spraying against service accounts

HypothesisAn attacker sprays passwords against AD service accounts slowly enough to stay under lockout thresholds. Service accounts often lack MFA and hold access to core business systems, so a hit is a direct path inward.
Data sourceAD authentication events, SIEM correlation, cloud access broker logs
MethodSequence failed logons by target account within a rolling window; flag single sources touching several service accounts while staying under the lockout threshold, then correlate the spray source against any later successful logon on a different account.
BaselineOne or two failed attempts per account from a known source inside a maintenance window is normal — anything hitting multiple service accounts outside those windows is not.
Documented
T1539Steal Web Session Cookie

Session-cookie theft bypassing MFA

HypothesisAn attacker steals an active session cookie via a browser-based infostealer after a phishing click and reuses it, bypassing MFA entirely — defeating the primary control the business relies on for account protection.
Data sourceEmail security gateway, endpoint telemetry, identity provider session logs, network metadata
MethodFlag browser processes spawning script hosts with network-bound arguments, then pivot on session tokens reused across multiple device fingerprints or geographies within a single token lifetime.
BaselineBrowser-to-script execution is rare outside admin tooling; a session token should stay on one device and location for its lifetime.
Documented
T1003.001LSASS Memory

Credential dumping via LSASS access

HypothesisAn unrecognized or unsigned process reads LSASS memory to extract cached credentials without tripping AV signatures. LSASS holds credentials for every logged-in user, privileged accounts included.
Data sourceEndpoint process-access telemetry, behavioral prevention alerting
MethodBaseline which processes legitimately open handles to LSASS and with which access masks, then flag any unlisted or unsigned process requesting high-privilege access. Where raw handle-access telemetry wasn't available, the hunt was documented as telemetry-blocked rather than reported clean — and the collection gap was raised as its own remediation item.
BaselineOnly allow-listed security agents touch LSASS with benign masks.
Documented
T1003.006DCSync

AD replication abuse for hash extraction

HypothesisA non-domain-controller account requests AD replication data, effectively extracting password hashes for arbitrary users.
Data sourceDirectory service access auditing, allow-list of known DCs and sync service accounts
MethodAlert on replication-related directory access from any account outside the approved allow-list, excluding machine accounts. Near-zero false positive once the allow-list is confirmed.
BaselineOnly DCs and one approved sync account ever request replication.

T1207Rogue Domain Controller

DCShadow — rogue DC registration

HypothesisAn attacker registers a rogue DC object to push malicious AD replication changes without generating a normal DC audit trail.
Data sourceAD configuration-partition change events, DC-to-DC replication traffic metadata
MethodWatch for transient directory-service objects appearing in the configuration partition, and pivot on the network side for replication protocol traffic sourced from anything that isn't a registered DC — the stronger of the two signals.
BaselineReplication traffic only ever flows between registered DCs.

Data sourceEndpoint registry-modification telemetry

Encoded and obfuscated PowerShell execution

HypothesisPowerShell is being used locally to execute encoded or obfuscated commands consistent with post-exploitation tooling.
Data sourceScript block and module logging, process-creation events

T1021.006WinRM / PS Remoting

Lateral movement over PowerShell remoting

HypothesisAn attacker establishes a remoting session to move laterally between hosts.
Data sourceNetwork logon events tied to the remoting host process, PowerShell logs, network traffic pivot
MethodFlag network logons whose process context is the remoting service host, then pivot on remoting traffic between hosts that don't normally talk to each other.

T1552.004 · T1003.002Credential Stores

DPAPI backup key & SAM database extraction

HypothesisAn attacker extracts the domain DPAPI backup key (decrypting protected secrets domain-wide), or accesses the registry keys and hives needed to decrypt the local account database offline.
Data sourceLSA and registry access auditing, process command-line telemetry
MethodAlert on non-system processes touching the security policy secrets and account hive paths, and correlate against hive-save command patterns; treat any backup-key retrieval outside a documented recovery exercise as an immediate escalation.
BaselineSystem-context access only, and never outside a documented DR exercise.

T1047 · T1546.003WMI

WMI remote execution & event-subscription persistence

HypothesisAn attacker uses WMI's remote process-creation method to execute code without dropping a binary, and/or registers a permanent event subscription (filter + consumer + binding) to persist and trigger on system events.
Data sourceWMI activity logging, process-creation events, RPC traffic pivot
MethodAllow-list known management tooling sources, then flag any new source host invoking remote process creation, plus any event consumer not tied to known tooling. Script-based event consumers are treated as an immediate escalation — they're rarely legitimate in ad-hoc form. WMI activity logging is off by default, so confirming it was enabled came before the hunt was trusted.
BaselineOnly known management/monitoring tooling uses remote WMI or registers subscriptions.
Documented
T1543.003 · T1021.002Services / SMB

Remote service creation (PsExec-style)

HypothesisA service is created remotely on a target host through the service control manager — a classic lateral-movement pattern — or installed locally with a suspicious binary path.
Data sourceService-installation events correlated with network logons, share-access auditing, SMB/RPC traffic pivot
MethodCorrelate service installation with a network logon from a remote source inside a tight window, flag binary paths outside approved deployment locations, and watch for handles opened against the service control manager over the admin share as a precursor step.
BaselineServices arrive via approved deployment tooling from known source IPs only.

T1574.001 · T1574.002DLL Side-Loading

DCOM- and WMI-triggered DLL side-loading

HypothesisAn attacker plants a malicious DLL in a non-standard path so it side-loads when a DCOM object is activated or when WMI activity occurs on the target, yielding remote code execution.
Data sourceImage/module load telemetry, DCOM activation events, RPC pivot
MethodFlag known-abusable DLL names loading from anywhere other than their expected system path, scoped to the process contexts that a DCOM or WMI launch would produce. Path-based logic keeps this low-FP.
BaselineThese DLLs only ever load from their canonical system directories.

Multi‑tacticComposite TTP Chain

Regional threat-actor composite hunt

HypothesisA regionally active, state-linked actor known for targeting financial institutions is present in the environment — using macro-laden spearphishing, living-off-the-land execution, legitimate remote-management tooling for C2, and browser/file credential theft. The chain is hunted as a whole, independent of any specific IOC match, precisely because the actor leans on signed and legitimate tooling to slip past signature-based controls.
Data sourceEndpoint process, script, module-load and registry telemetry; proxy logs; email security; SIEM correlation; threat intel platform
MethodRun each link of the chain as its own query — office applications spawning script hosts, obfuscated interpreter commands, unapproved remote-management agents, non-browser processes reading browser credential stores, bursts of discovery commands on a single host — then chain-correlate them in the SIEM. One indicator is suspicious; a chained match on the same host in a short window is high-confidence. Beacon-like traffic to rare or newly-seen domains at consistent intervals is the network-side confirmation.
BaselineOffice apps never spawn script hosts; remote-management tools only arrive via ticketed deployment; discovery commands appear individually rather than in tight bursts; outbound traffic shows normal jitter to well-known domains.
Documented
Per‑CVEEdge Appliance Exploitation

Post-exploitation hunt on emerging VPN/edge CVEs

HypothesisAn internet-facing VPN or edge appliance has already been exploited via an emerging CVE — patch status alone proves nothing about what happened before the patch landed.
Data sourceEndpoint and network telemetry, appliance logs, SIEM correlation
MethodScoped at the start of each cycle to the CVE in play. Deliberately targets post-exploitation artifacts — web shells, unexpected outbound connections, newly created local accounts — rather than verifying patch level. A patched appliance with no artifacts and confirmed telemetry closes as a documented baseline; patch status by itself is never accepted as evidence.
BaselineNo post-exploitation artifacts present, regardless of patch state.
