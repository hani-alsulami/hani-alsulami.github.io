My hunting playbook

The process I run my team through on every hunt — from a hunch to a shipped detection. Numbered because the order matters: skip a step and the finding won't survive review.

01

Form the hypothesis

Start from a specific, falsifiable claim tied to attacker behavior — not "look for anomalies." Ground it in a technique we haven't proven we'd catch, a gap flagged by recent threat intel, or a near-miss from a real incident.

MITRE ATT&CK Navigatorthreat intel feedspast incident reviews
02

Map the data

Identify exactly which log sources and fields would prove or disprove the hypothesis, and confirm they're actually being collected — with enough retention and fidelity — before writing a single query.

SysmonEDR telemetryZeek/network logsSIEM index catalog
03

Build and scope the query

Write the narrowest query that could surface the behavior, then widen deliberately. Every hunt starts scoped to a small time window and a known-risky asset group before going environment-wide.

SPL / KQLSigma rulesJupyter for ad-hoc analysis
04

Validate against ground truth

Test the query against known-good activity and, where possible, a controlled simulation of the technique, so a hit means something before it ever reaches an analyst.

Atomic testsbaseline traffic samplespeer review
05

Document the finding

Every hunt gets a short record regardless of outcome: hypothesis, data used, result, and false-positive rate. A hunt that found nothing is still worth logging — it closes a gap in what we've verified.

internal wikiATT&CK technique tagging
06

Hand off to detection engineering

A hunt that proves valuable doesn't stay a manual query — it graduates into a tuned, owned detection rule with a documented response playbook, so the next occurrence is caught automatically, not re-discovered.

correlation rulesSOAR playbookson-call runbook
